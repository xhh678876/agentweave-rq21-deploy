# Task: Implement Batch Report Export Pipeline with Background Workers

## Background

Celery (https://github.com/celery/celery) is a distributed task queue for Python. The project needs a new batch report export pipeline that allows API callers to submit export requests, receive a job ID immediately, and poll for completion. Reports are generated by aggregating data from multiple sources — a process that can take 30–120 seconds — and must be processed by background workers rather than blocking the HTTP response. The pipeline must handle worker crashes, duplicate deliveries, and partial failures gracefully.

## Files to Create/Modify

- `celery/app/batch_export.py` (new) — Background task definitions for report generation, including the main export task, a cleanup task for expired jobs, and a chain that composes extraction → transformation → upload steps
- `celery/app/job_store.py` (new) — Job state persistence layer: create, read, and update job records with status transitions (pending → running → succeeded / failed), timestamps, and error details
- `t/unit/app/test_batch_export.py` (new) — Unit tests for the export pipeline covering retry behavior, idempotency, state transitions, timeout enforcement, and dead-letter handling

## Requirements

### Task Definitions (`batch_export.py`)

- Define a Celery task `generate_report` decorated with `@app.task(bind=True)` that accepts `job_id: str` and `params: dict` (containing `report_type`, `date_range`, `filters`)
- `generate_report` must transition the job to `RUNNING` before starting work, and to `SUCCEEDED` or `FAILED` upon completion
- Configure `max_retries=3`, `task_time_limit=300`, `task_soft_time_limit=240`, and `task_acks_late=True` on the task
- On `SoftTimeLimitExceeded`, the task must mark the job as `FAILED` with error message `"Export timed out after 240s"` and not retry
- On `ConnectionError` or `IOError`, the task must retry with exponential backoff: delay = `2 ** self.request.retries * 30` seconds
- On `ValueError` (e.g., invalid `report_type`), the task must fail immediately without retry and record the validation error
- Define a second task `cleanup_expired_jobs` that deletes all jobs older than 72 hours in `SUCCEEDED` or `FAILED` status
- Define a task chain `run_export_pipeline` that composes three subtasks: `extract_data.s(job_id)` → `transform_data.s()` → `upload_result.s(job_id)`, where each subtask is a separate `@app.task`

### Job State Management (`job_store.py`)

- Implement a `JobStore` class with methods: `create(job_id, params) -> Job`, `get(job_id) -> Job | None`, `update_status(job_id, status, **fields) -> None`, `list_by_status(status, limit=50) -> list[Job]`, `delete_expired(max_age_hours=72) -> int`
- `Job` is a dataclass with fields: `id: str`, `status: str`, `params: dict`, `created_at: datetime`, `started_at: datetime | None`, `completed_at: datetime | None`, `result: dict | None`, `error: str | None`, `attempts: int`
- `update_status` must automatically set `started_at` when transitioning to `RUNNING` and `completed_at` when transitioning to `SUCCEEDED` or `FAILED`
- `update_status` must reject invalid transitions: once a job is `SUCCEEDED` or `FAILED`, it cannot transition to any other state; must raise `InvalidTransitionError`
- `create` must raise `DuplicateJobError` if a job with the same `id` already exists

### Idempotency

- `generate_report` must check job status before starting work; if the job is already `SUCCEEDED`, it must return immediately without re-processing
- If the job is already `RUNNING` (from a previous attempt that crashed), the task must continue processing rather than raising an error
- Each call to an external upload service within `upload_result` must include an idempotency key derived from the `job_id` to prevent duplicate uploads on retry

### Dead Letter Handling

- After exhausting all 3 retries, `generate_report` must write a record to a `dead_letter` list containing: `task_name`, `job_id`, `params`, `error`, `attempts`, and `failed_at` ISO timestamp
- The dead-letter record must be retrievable via `JobStore.get_dead_letters(limit=100) -> list[dict]`

### Expected Functionality

- Calling `generate_report.delay(job_id="rpt-001", params={"report_type": "sales", "date_range": ["2024-01-01", "2024-01-31"]})` → job transitions pending → running → succeeded; `JobStore.get("rpt-001").status == "SUCCEEDED"`
- `generate_report` called twice with the same `job_id` where job is already `SUCCEEDED` → second call returns immediately, `attempts` count does not increment
- Task encounters `ConnectionError` on first two attempts, succeeds on third → job shows `attempts=3`, status `SUCCEEDED`, backoff delays were 30s and 60s
- Task encounters `ValueError("Unknown report_type: xyz")` → job immediately `FAILED`, no retry, `error` field contains the message
- `SoftTimeLimitExceeded` raised during processing → job `FAILED` with timeout message, no retry
- `cleanup_expired_jobs` with 5 completed jobs older than 72h and 2 within 72h → returns 5, only 2 remain
- `run_export_pipeline` where `transform_data` raises an exception → upstream `extract_data` result is discarded, `upload_result` never executes
- `update_status("rpt-001", "RUNNING")` on a job already in `SUCCEEDED` → raises `InvalidTransitionError`
- After 3 failed retries → dead-letter list contains one entry with correct `job_id`, `error`, and `failed_at`

## Acceptance Criteria

- `generate_report` task is importable from `celery.app.batch_export` and correctly decorated with `bind=True`, retry, and timeout settings
- Job state transitions are persisted and queryable via `JobStore`; invalid transitions raise `InvalidTransitionError`
- Retries use exponential backoff only for transient errors (`ConnectionError`, `IOError`); permanent errors (`ValueError`) and timeouts fail immediately
- Idempotent re-execution: re-running a task for an already-succeeded job produces no side effects
- Dead-letter records are written after retry exhaustion and contain all required fields
- Task chain `run_export_pipeline` short-circuits on subtask failure
- `cleanup_expired_jobs` correctly filters by age and terminal status
- All tests in `t/unit/app/test_batch_export.py` pass via `python -m pytest t/unit/app/test_batch_export.py -v`
